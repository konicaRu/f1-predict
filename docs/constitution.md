# Конституция проекта F1 Predict

Короткий свод **незыблемых принципов** проекта. На него сверяется каждая спека, план,
реализация и ревью. Это консолидация уже принятых решений — детали живут в
[`docs/plan.md`](plan.md) (единый источник правды) и [`CLAUDE.md`](../CLAUDE.md).
Правило меняется только осознанно: сначала правим здесь, потом код.

Приоритет при конфликте: **явная просьба пользователя → эта конституция → скиллы Superpowers → дефолт.**

---

## 1. Продукт и правила игры

- **Слепой прогноз.** Дедлайн — четверг, фиксированное время по МСК, до квалификации.
  Чужие прогнозы скрыты до дедлайна.
- **Формула очков неизменна:** вес ЗАЯВЛЕННОЙ позиции Y из `[25,18,15,12,10,8,6,4,2,1]`,
  штраф `2·|X−Y|`, бонус `+3` за точное попадание, минимум 0. Тайбрейкер: точные попадания → лучшая гонка.
  Референс-реализация — `CLAUDE.md` / `src/lib/scoring.ts`.
- **Один пилот = один слот.** Валидируется И в UI, И на сервере (триггер БД) — не только в интерфейсе.
- **Старт зачёта:** прошлые гонки — демо (`scored=false`), зачёт идёт с первой готовой гонки.

## 2. Безопасность (RLS-first)

- **Anon-ключ публичен → вся безопасность в БД (RLS).** Никакой защиты «на уровне UI» — только политики Postgres.
  Скрытие чужих прогнозов, гейт членства, проверка дедлайна — всё через RLS/триггеры.
- **Дедлайн проверяется на сервере** (политика/триггер), не доверяем клиенту.
- **Секреты НИКОГДА в git.** Публичны и допустимы в бандле: anon/publishable-ключ, Supabase URL.
  Секретны и только вне репо: пароль БД (`.env`), `service_role`, Telegram-токен. `.gitignore` исключает все `.env*`.

## 3. Данные и время

- **Время: хранить в UTC, показывать в МСК.** Всегда.
- **Источник данных F1 — Jolpica API** (истина); хардкод грида в плане — спекулятивный, игнорируется.
  Фолбэк — OpenF1.
- **Пул пилотов фиксируется снимком на открытие гонки** (`race_driver_pool`), прогноз валидируется против снимка.
- **Занос результата двухфазный:** T+4ч provisional → T+24ч final, плюс ручной override. Триггеры/итоги — на `final`.

## 4. Архитектура и инфраструктура

- **Сервис работает без включённого компа разработчика.** Прод = статика на GitHub Pages + Supabase-облако.
  Комп нужен только для разработки/деплоя.
- **Cloud-direct.** Локальный Docker-стек Supabase на этой машине НЕ работает (сеть Docker-VM). Работаем
  напрямую против облака: миграции/тесты через `scripts/db/` (transaction-пулер `:6543`), БЕЗ supabase migration-трекинга.
- **Keep-alive обязателен** — free-Supabase засыпает после ~7 дней недостаточной активности БД.
  Таймер обнуляет РЕАЛЬНЫЙ запрос к БД (не пустой health-пинг), и нужна активность каждый день, не 1-2/нед.
  GitHub Actions **дважды в день** (scheduled-джобы GitHub ненадёжны → берём частотой). Инцидент 2026-07-14: уснул на ×2/нед.
- **Тайминг на `pg_cron`**, не на GitHub Actions (точнее): напоминания, автозанос результатов.
- **Деплой автоматический** — push в `main` → GitHub Actions собирает Vite и публикует на Pages
  (VITE-переменные из GitHub-секретов, SPA-фолбэк `404.html`).
- **Сеть до Supabase флапает → устойчивость к обрывам:** ретрай транзиентных сбоев (и на бэкенде в `scripts/db`,
  и во фронтовом `db.ts`), но НЕ ретраим ошибки БД/RLS.

## 5. Процесс разработки

- **Каждый под-проект проходит цикл:** brainstorming → спека (`docs/superpowers/specs/`) → план
  (`docs/superpowers/plans/`, bite-sized задачи) → реализация → merge. Скиллы Superpowers + `karpathy-guidelines`.
- **Гейты качества сабагентов:** на каждую задачу — свежий сабагент-исполнитель + **spec-compliance-ревью** +
  **code-quality-ревью**; перед merge — **финальное ревью всей ветки** (в т.ч. проверка покрытия: код ↔ спека, без дрейфа).
- **TDD на бэкенде** (SQL/функции): тест сначала, падает, потом реализация (pg-тесты в `scripts/db`).
- **Ветки и коммиты:** работа на feature-ветке, частые осмысленные коммиты, `git save`/`git load` — конвенции сессии.
  Финал — через `finishing-a-development-branch`.
- **Журнал `MEMORY.md`** ведём всегда: статус, открытые вопросы/бэклог, лог сессий (даты — абсолютные).

## 6. Код и зависимости

- **YAGNI и минимум зависимостей.** Не тащим фреймворк/пакет, пока он не оправдан масштабом (лига ~10 человек).
- **Язык:** общение и доки — русские, код и идентификаторы — английские.
- **Mobile-first** для игровых экранов (напр. прогноз — боковая раскладка на узком экране, см. `plan.md` §16.11).
- Читать код в стиле окружения: те же идиомы, именование, плотность комментариев.
