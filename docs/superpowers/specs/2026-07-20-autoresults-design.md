# Автозабор результатов гонки (дизайн)

Дата: 2026-07-20
Статус: согласован, готов к плану
Сверено с `docs/constitution.md` (§2 RLS/секьюрити — не применимо, скрипт trusted-серверный; §4
инфраструктура — GitHub Actions, то же расширенное исключение, что и Telegram; §5 процесс; §6 YAGNI).
Дополняет уже открытую (не влитую) ветку `telegram-notify` — переиспользует её `sendTelegram()`,
workflow и расписание вместо отдельной ветки.

## 1. Цель

Автоматически заносить результат гонки после того, как она прошла — сейчас единственный способ занести
результат ЭТО ручной ввод через Админку, и триггер для «пора занести» не было ничем не подсказан.
Повод: Гран-при Бельгии (round 10) прошёл в воскресенье 2026-07-19, к понедельнику 2026-07-20 результата
всё ещё нет — руками занести могли, но узнать, что «пора», можно было только зайдя на сайт самостоятельно.

## 2. Контекст (что уже проверено)

- **Основной источник — Jolpica**, эндпоинт `2026/{round}/results` (Ergast-совместимый). Уже используется
  в `scripts/import/import.js`'s `importResults()` для разовой исторической загрузки (демо-гонки 1-9):
  запрос отдаёт неполный/пустой `Results`, пока гонка официально не классифицирована; готовность
  проверяется условием `Results.length >= 10`. `Driver.driverId` — тот же slug, что наш `drivers.id`
  (`hamilton`, `norris`, ...) — прямое совпадение, без маппинга.
- **Резервный источник — OpenF1**, проверен вживую (curl) в ходе брейншторма:
  - `GET /v1/sessions?country_name={страна}&year=2026&session_name=Race` → `session_key` (сессия гонки).
  - `GET /v1/session_result?session_key={key}` → массив `{position, driver_number, dnf, dns, dsq, ...}`.
  - `GET /v1/drivers?session_key={key}` → массив `{driver_number, name_acronym, ...}` — `name_acronym`
    совпадает с нашим `drivers.code` (3 буквы, `VER`/`HAM`/...).
  - Лимит free-тарифа: 3 req/s, 30 req/min — на 1 гонку нужно ~3 запроса, лимит не проблема.
  - Сопоставление сессии с нашей гонкой — **по дате**, не по названию (надёжнее парсинга англ. названий
    стран из `races.name`): среди `sessions?year=2026&session_name=Race` берём ту, чей `date_start`
    ближе всего к нашему `races.race_datetime_utc` (в пределах разумного окна, напр. ±3 дня).
- `set_race_result(p_race_id, p_positions, p_reason)` (миграция `0009_admin_results.sql`) — тот же RPC,
  что вызывает Админка: гейт админа (пропускает direct-подключение без `auth.uid()` — наш скрипт
  подключается напрямую к Postgres, как `scripts/db`/`scripts/import`/`scripts/export`/`scripts/telegram`,
  так что гейт не блокирует), валидация 10-из-пула, журнал `result_changes`, `status='resulted',
  scored=true`. Это единственный путь заноса результата — используем его, не пишем в `results` напрямую
  (в отличие от одноразового `importResults()` из Фазы 1, который бил в таблицу в обход RPC — так делать
  для боевого автозабора нельзя, RPC даёт валидацию+журнал+`scored`, которых напрямую нет).
- `scripts/telegram/lib.js` (ветка `telegram-notify`, уже принята Task 1-6) — `sendTelegram()`,
  `q()`/`close()` с тем же `.env`-паттерном. Переиспользуем `sendTelegram` напрямую (`require`
  относительным путём из соседней папки) вместо копирования.
- `.github/workflows/telegram-notify.yml` — уже есть cron `0 */2 * * *` (сейчас под режим `results` —
  проверка неотправленных объявлений) и `0 7 * * 1` (`raceweek`, Пн 10:00 МСК).

## 3. Решения (из брейншторма)

- **Дополняем ветку `telegram-notify`**, не заводим отдельную — экономит дублирование Telegram-клиента
  и избегает зависимости между двумя не влитыми ветками.
- **Единственная фаза — сразу `final`.** Без provisional/пересчёта: нашли результат → занесли как
  финальный. Если после гонки что-то поменяется (пенальти/дисквалификация) — те же руки, что и сейчас,
  правят через существующий override в Админке (уже поддерживает журнал изменений).
- **Порядок источников:** Jolpica → если пусто → OpenF1 → если и там пусто → ничего не делаем,
  следующая попытка через 2 часа (тот же cron, что уже есть).
- **Напоминание при неудаче обоих источников — в Telegram** (не баннер на сайте): переиспользует уже
  построенный канал, точно увидится, не требует захода на сайт. Отправляется по понедельникам в 10:00
  МСК (тот же слот, что и `raceweek`) — если есть гонка с `race_datetime_utc` в прошлом и всё ещё
  `status='open'`. Повторяется каждый следующий понедельник, пока не занесено (это осознанно —
  не одноразовое, чтобы не забылось на неделю).

## 4. Архитектура и файлы

```
scripts/autoresults/
  package.json      — зависимости: pg (fetch — встроен в Node ≥18)
  lib.js            — pg-подключение (тот же паттерн, что scripts/telegram/lib.js)
  jolpica.js        — fetchJolpicaResults(round): string[] | null (топ-10 driverId или null, если не готово)
  openf1.js         — fetchOpenF1Results(raceDatetimeUtc, driverCodeToId): string[] | null
  fetch.js          — точка входа: находит просроченные open-гонки, пробует Jolpica→OpenF1,
                      заносит через set_race_result(), возвращает сводку
.github/workflows/telegram-notify.yml
  — расширяется: на cron `0 */2 * * *` сначала запускать `node scripts/autoresults/fetch.js`,
    потом уже существующий `node scripts/telegram/notify.js results` (цепочка: нашли→занесли→объявили)
  — на cron `0 7 * * 1` (Пн, уже `raceweek`) дополнительно запускать `node scripts/telegram/notify.js remind`
scripts/telegram/notify.js
  — новый режим `remind` (4-й наряду с raceweek/deadline/results): ищет open-гонки с
    race_datetime_utc в прошлом, шлёт сообщение-напоминание
```

**Границы:** `scripts/autoresults/lib.js` — только подключение к БД (без Telegram — автозабору не нужно
слать сообщения, этим занимается уже существующий `results`-режим `notify.js`, который сработает следом
в том же прогоне workflow). `jolpica.js`/`openf1.js` — по одной чистой функции на источник, каждая
возвращает либо массив из 10 driver-id (в порядке 1→10), либо `null` (источник ничего не нашёл/гонка ещё
не классифицирована) — никакой сторонней логики (без записи в БД внутри них, только фетч+трансформация).
`fetch.js` — оркестрация: находит гонки, пробует источники по очереди, вызывает `set_race_result()`.

## 5. `jolpica.js`

```
async function fetchJolpicaResults(round) → string[] | null
```
- `GET https://api.jolpi.ca/ergast/f1/2026/{round}/results.json?limit=100`
- Если `MRData.RaceTable.Races[0].Results.length < 10` → `null` (не классифицирована).
- Иначе — сортировка по `position` (числом), топ-10, `map(r => r.Driver.driverId)`.

## 6. `openf1.js`

```
async function fetchOpenF1Results(raceDatetimeUtc, driverCodeToId) → string[] | null
```
- `GET https://api.openf1.org/v1/sessions?year=2026&session_name=Race` → найти сессию с `date_start`
  ближайшим к `raceDatetimeUtc` (окно ±3 дня; если ничего в окне — `null`, гонки в OpenF1 ещё нет).
- `GET https://api.openf1.org/v1/session_result?session_key={key}` → если `< 10` строк с валидной
  `position` (не `dnf`/`dns`/`dsq` не считаются финишем, но НЕ исключаются из топ-10 подсчёта — их
  `position` в OpenF1 всё равно отражает итоговую классификацию FIA, включая сошедших с зачётным
  результатом) → `null`.
- `GET https://api.openf1.org/v1/drivers?session_key={key}` → карта `driver_number → name_acronym`.
- Сопоставить `name_acronym` (= `drivers.code` в нашей БД) → наш `drivers.id` через переданный
  `driverCodeToId` (карта code→id, строится в `fetch.js` одним запросом `select id, code from drivers`
  и передаётся в обе функции-источника, чтобы не дублировать запрос).
- Отсортировать по `position`, взять топ-10, замаппить через `driverCodeToId`.

## 7. `fetch.js`

```
async function main()
```
- `select id, round, name, race_datetime_utc from races where status='open' and race_datetime_utc < now()`
- Карта `code→id` пилотов — один запрос, переиспользуется для всех найденных гонок.
- На каждую гонку по очереди: `fetchJolpicaResults(round)` → если `null`, `fetchOpenF1Results(...)` →
  если оба `null`, пропустить (следующая попытка через 2 часа). Если что-то нашли — вызвать
  `set_race_result(race.id, positions)` (через прямой `q()`-вызов RPC, без причины/reason — это не
  правка, а первый занос).
- Печатает сводку в консоль (`autoresults: <гонка> — занесено (Jolpica|OpenF1)` /
  `autoresults: <гонка> — источники пока пусты`), как остальные `scripts/*`.

## 8. Режим `remind` в `notify.js`

- `select id, round, name from races where status='open' and race_datetime_utc < now()`
- Если пусто — `console.log('remind: просроченных гонок нет')`, ничего не шлём.
- Иначе на каждую — сообщение:
  ```
  ⚠️ Автопоиск не нашёл результат <b>{name}</b> — занеси вручную в Админке.
  {SITE_URL}/admin
  ```
- Без трекинга «уже напомнили» — сознательно шлётся каждый понедельник, пока гонка остаётся `open`.

## 9. Workflow — порядок шагов

На cron `0 */2 * * *` (было: только `results`) — теперь два шага подряд в одном job:
1. `node scripts/autoresults/fetch.js` (может занести результат → `status` гонки становится `resulted`)
2. `node scripts/telegram/notify.js results` (увидит только что занесённую гонку, объявит, если ещё не
   объявлена — использует уже существующий `telegram_announced_at`)

На cron `0 7 * * 1` (было: только `raceweek`) — теперь два шага подряд:
1. `node scripts/telegram/notify.js raceweek`
2. `node scripts/telegram/notify.js remind`

Секреты не меняются (те же `SUPABASE_DB_URL`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` — `autoresults`
не обращается к Telegram, только к БД+внешним API, поэтому доп. секретов не требует).

## 10. Тестирование

- **Юнит-уровень через `scripts/db/runner.js`:** нельзя протестировать HTTP-фетчеры изолированно без
  живого запуска — вместо этого сверяем SQL-запрос «просроченные open-гонки» напрямую, как в предыдущих
  задачах этого проекта.
- **Реальный смоук — Бельгия (round 10) уже просрочена**, отдельная фикстура не нужна: после реализации
  просто запускаем `node scripts/autoresults/fetch.js` локально (с `.env`, где уже есть `SUPABASE_DB_URL`)
  и смотрим, нашёлся ли результат через Jolpica (гонка реально прошла 2026-07-19, к моменту реализации
  Jolpica должен её уже классифицировать). Если Jolpica почему-то пуст — тест пути OpenF1 тоже пройдёт
  естественно (тот же реальный вызов). Если результат Бельгии по какой-то причине всё ещё не
  классифицирован НИ ГДЕ на момент теста — это будет означать, что источники реально ничего не нашли,
  и тогда нужно будет либо подождать, либо ты заносишь Бельгию вручную (как договаривались раньше) —
  не блокирует мерж остальной автоматики.
- Проверить `remind`-режим отдельно: временно (без правки БД) можно просто убедиться, что SQL-запрос
  возвращает Бельгию, пока она `open` — если к моменту теста автозабор её уже занёс, `remind` должен
  вернуть пустой список (проверка на другой ситуации невозможна без фикстуры; фикстуру заводить не
  будем — риск несоразмерен пользе для строки из одного SQL-запроса, уже покрытого паттерном из
  `results`-режима).

## 11. Вне скоупа

- Provisional-статус / пересчёт после пенальти — правки остаются ручными через существующий override.
- Автосинк пилотов (упомянут в старом плане Фазы 4) — отдельная, не запрошенная сейчас задача.
- Уведомление в Telegram при УСПЕШНОМ автозаносе с отдельным текстом «занесено автоматически» — не
  нужно: обычное `results`-сообщение (уже есть) не различает, кто занёс, ручной ввод или автозабор, и
  это не требовалось.
- Ретраи внутри `fetch.js` при сетевых сбоях Jolpica/OpenF1 — не нужны, тот же cron через 2 часа и есть
  ретрай на уровне расписания (YAGNI, как и `scripts/telegram`).
