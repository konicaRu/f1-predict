# Сброс пароля (дизайн)

Дата: 2026-07-15
Статус: согласован, готов к плану
Сверено с `docs/constitution.md` (§2 RLS/Auth не требуется отдельно — используется штатный Supabase Auth, §6 YAGNI, §7 дизайн).

## 1. Цель

Дать участнику лиги восстановить доступ, если он забыл пароль — сейчас (после Фазы 2a) восстановления
нет вообще, только ручной сброс админом через дэшборд Supabase. Принято требованием ещё 2026-06-30/
2026-07-07, отложено до появления времени — актуально сейчас, когда MVP живой и друзья реально играют.

## 2. Контекст (что уже готово)

- Supabase Auth: email+пароль, «Confirm email» выключен, email-сигнапы включены.
- `src/lib/supabase.ts`: клиент с `persistSession: true, autoRefreshToken: true` — `detectSessionInUrl`
  не задан явно → используется дефолт `true` (SDK сам вытаскивает токен из URL и поднимает сессию).
- `src/pages/Login.tsx` — форма email+пароль, паттерн `auth-card`, ошибки через `.auth-err`.
- `src/App.tsx` — публичные маршруты `/login`, `/signup`, `/redeem` вне `ProtectedRoute`.
- Стили `.auth-card` и сопутствующие — уже есть, переиспользуем.

## 3. Решения (из брейншторма)

- **Подход A — штатный флоу Supabase Auth** (`resetPasswordForEmail` → recovery-сессия по ссылке →
  `updateUser({password})`). Без своего бэкенда/Edge Function — YAGNI для лиги из нескольких друзей.
- **UI входа** — инлайн-форма (email + кнопка) разворачивается на `/login` под полем пароля по ссылке
  «Забыли пароль?», отдельный маршрут не заводим.
- **После смены пароля** — автовход (recovery-сессия уже валидна) + редирект на `/calendar`. Не просим
  логиниться повторно.
- **Сообщение после отправки письма** — всегда одинаковое, не выдаёт существование email в системе
  (стандартная практика; закрытая лига, список участников не должен палиться перебором).
- **Невалидная/истёкшая ссылка на `/reset`** — явное сообщение «Ссылка недействительна» + ссылка на `/login`.

## 5. Экран Login — инлайн-форма сброса

- Под полем пароля — ссылка `<button type="button">Забыли пароль?</button>` (не submit, чтобы не
  триггерить форму входа).
- Клик — локальный state `forgotOpen: boolean` переключает видимость: вместо кнопки «Войти» показываем
  email-инпут (предзаполненный из основного поля email, если оно уже введено) + кнопку «Отправить ссылку».
- Отправка: `supabase.auth.resetPasswordForEmail(email, { redirectTo: <origin>/f1-predict/reset })`.
  `<origin>` берём из `window.location.origin` (работает и на GitHub Pages, и локально на dev-сервере).
- Результат (успех ИЛИ ошибка сети/Supabase — за исключением явно неверного формата email, что ловится
  HTML5 `type="email" required` до сабмита) — один и тот же текст: «Если такой email зарегистрирован,
  письмо со ссылкой на сброс отправлено». Кнопка блокируется на время запроса (`busy`), как в остальных
  формах проекта.
- Ссылка «Назад ко входу» возвращает `forgotOpen=false`.

## 6. Экран `/reset` (`ResetPassword.tsx`)

- Публичный маршрут (вне `ProtectedRoute` — обычная auth-сессия здесь не нужна, нужна recovery-сессия,
  которую Supabase SDK поднимает сам при загрузке страницы по ссылке из письма).
- При монтировании: `supabase.auth.getSession()` — если сессии нет (прямой заход, истёкшая/использованная
  ссылка) → состояние `invalid`: «Ссылка недействительна или уже использована» + `<Link to="/login">`.
- Если сессия есть → форма: «Новый пароль» + «Повторите пароль» (клиентская проверка совпадения перед
  сабмитом, без отдельного бэкенд-запроса на валидацию).
- Сабмит → `supabase.auth.updateUser({ password })`. Успех → редирект на `/calendar` (сессия уже активна,
  `AuthProvider` подхватит её через существующий `onAuthStateChange`). Ошибка (напр. пароль короче
  минимума Supabase) → показываем `error.message` в `.auth-err`, форма остаётся.
- Loading-состояние на время проверки сессии при монтировании (короткий «…», сессия локальная — сети
  почти не требует, но `getSession()` может дождаться `autoRefreshToken`).

## 7. Тестирование

- **Фронт-верификация:** `npm run build` зелёный.
- **Ручной e2e-смоук в браузере:**
  1. На `/login` нажать «Забыли пароль», ввести свой email (`prokol35@gmail.com`), отправить — увидеть
     единый текст об отправке.
  2. Проверить письмо (free-SMTP Supabase может уйти в спам — проверить обе папки).
  3. Перейти по ссылке → должен открыться `/reset` с формой нового пароля (не «Ссылка недействительна»).
  4. Задать новый пароль → редирект на `/calendar`, залогинен.
  5. Разлогиниться, зайти на `/login` новым паролем — убедиться, что сработало.
  6. Отдельно: зайти на `/reset` напрямую (без перехода по письму) — убедиться, что показывает «Ссылка
     недействительна» + ссылку на `/login`.
- **Fallback, если письмо не пришло за разумное время:** сверить в Supabase дэшборде (Auth → Users →
  Email Templates/Logs), что запрос ушёл; если проблема с доставкой — фиксировать отдельно в беклоге
  (свой SMTP), не блокирует эту фазу — механизм на нашей стороне работает корректно независимо от
  доставки письма конкретным провайдером.
- Автотестов на этот флоу нет (как и для остального auth в проекте) — только ручной e2e.

## 8. Вне скоупа

- Свой SMTP (только если увидим реальные проблемы с доставкой в проде).
- Rate-limit UI поверх встроенного в Supabase (Supabase уже лимитирует `resetPasswordForEmail` сам).
- Уведомление админа о сбросах, аудит-журнал попыток сброса.
