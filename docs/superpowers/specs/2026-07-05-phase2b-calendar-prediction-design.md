# Фаза 2b — Календарь + Прогноз (дизайн)

Дата: 2026-07-05
Статус: согласован, готов к плану

## 1. Цель

Дать участнику лиги два ядровых экрана:
- **Календарь** — список гонок сезона со статусами, подсветкой ближайшей открытой, каунтдауном до дедлайна и меткой «прогноз сделан».
- **Прогноз** — расстановка топ-10 через **tap-to-assign** для открытой гонки, с серверной валидацией дедлайна/состава (уже готова в Фазе 0), сохранением и редактированием до дедлайна.

Плюс серверная функция `open_race()` для перевода гонки в статус `open` со снимком пула, и разовый dev-бутстрап, чтобы в 2b было что тестировать (Бельгия, round 10).

Витрина результатов, общий зачёт и drift-chart — **не** входят в 2b (Фаза 3). Полноценная админ-кнопка «открыть гонку» — 2c (в 2b только функция + dev-скрипт). Drag-and-drop — позже (полировка), сейчас tap-to-assign.

## 2. Контекст (что уже готово)

Backend полностью покрывает запись прогноза:
- `predictions(user_id, race_id, positions jsonb, unique(user_id, race_id))` — `positions` = массив из 10 `driver_id`, индекс 0..9 = слот 1..10.
- RLS (`0003_rls.sql`): своя запись видна всегда; чужая — только после дедлайна. Insert/update только своей строки и только пока `now() <= deadline_utc`. DELETE запрещён.
- Триггер `validate_prediction` (`0004_validation.sql`): ровно 10 разных пилотов, все из `race_driver_pool` гонки.
- `races.status ∈ ('demo','open','closed','resulted')`, `deadline_utc not null`.
- В облаке: 22 гонки, 22 активных пилота с командами/цветами, 8 демо-раундов (`status='demo'`, в прошлом).

Значит фронту 2b нужно: собрать корректный массив 10 `driver_id`, отправить upsert-ом, аккуратно отобразить серверные ошибки.

Сегодня 2026-07-05: дедлайн Британии (round 9) прошёл → тестовая открытая гонка = **Бельгия (round 10)**.

## 3. Решения (из брейншторма)

- **Навигация:** календарь — точка входа. Тап по открытой гонке → `/predict/:raceId`. Таб «Прогноз» без id редиректит на ближайшую открытую (или заглушка «нет открытых гонок»).
- **Не-открытые гонки:** upcoming (`demo`, дедлайн в будущем) — некликабельны, бейдж «скоро». Locked (`open`, дедлайн прошёл) — кликабельны в **read-only**. Результаты (`closed`/`resulted`) — некликабельны (витрина = Фаза 3).
- **Интеракция:** tap-to-assign (drag-and-drop отложен).
- **Открытие гонки:** SQL-функция `open_race()` + dev-бутстрап; полная админ-кнопка в 2c.
- **Фронт-архитектура:** подход A — тонкий типизированный `src/lib/db.ts` (единственное место вызовов `supabase.from` для данных лиги) + локальный `useState` в экранах + презентационные компоненты без обращения к БД. Без React Query / глобального стора (YAGNI для лиги ~10 человек).
- **Мобильная раскладка «Прогноз»:** боковая (слоты слева узкой колонкой, пул справа), обе колонки на одном экране (принятое требование, `plan.md` §16.11).

## 5. Экран «Календарь»

**Данные:** `listRaces()` (все гонки 2026, сорт по round) + `getMyPredictionRaceIds()` (где есть мой прогноз).

**Классификация гонки (status + дедлайн → вид):**

| Вид | Условие | Карточка |
|---|---|---|
| Открыта | `status='open'` И `now() ≤ deadline_utc` | кликабельна → `/predict/:id`; каунтдаун; бейдж ✓ если прогноз есть |
| Заблокирована | `status='open'` И `now() > deadline_utc` | кликабельна → `/predict/:id` (read-only) |
| Скоро | `status='demo'` И `deadline_utc` в будущем | некликабельна, приглушена, бейдж «скоро» |
| Прошедшая | `status='demo'` И дедлайн прошёл, ИЛИ `status ∈ ('closed','resulted')` | некликабельна, бейдж «результаты» (витрина = Фаза 3) |

**Группировка (визуальные блоки):** «Активные» (открытые, подсветка cyan-рамкой ближайшей) → «Ближайшие» (скоро) → «Прошедшие». Сортировка внутри — по round.

**RaceCard показывает:** round + название, дату гонки в МСК, статус-бейдж, каунтдаун (для открытых), метку «✓ прогноз» / «— нет прогноза».

**Роутинг:** новый `/predict/:raceId`. Таб «Прогноз» (`/predict`) → `nextOpenRace()`; если null — заглушка «Сейчас нет открытых гонок, смотри Календарь».

## 6. Экран «Прогноз»

**Данные:** `getRaceWithPool(raceId)` (гонка + пул из `race_driver_pool`) + `getMyPrediction(raceId)`.

**Состояние (локально):**
- `slots: (driverId | null)[]` длиной 10.
- Инициализация: сохранённый прогноз → по слотам; иначе все `null`.
- Пул делится на «доступные» (не в слотах) и «назначенные» (приглушены).

**Tap-to-assign:**
1. Тап по пилоту в пуле → первый пустой слот сверху.
2. Тап по занятому слоту → освобождает (пилот назад в пул).
3. Прицельный режим: тап пилота (подсветка «выбран») → тап пустого слота → в это место. Держим оба режима.
4. Один пилот = один слот (UI не даёт дубль; дублирует триггер).

**Сохранение:** кнопка «Сохранить» активна при всех 10 заполненных слотах → `savePrediction(raceId, ids)` (upsert). Успех → тост «Прогноз сохранён» + метка на календаре. Прогноз редактируем до дедлайна (RLS разрешает update).

**Раскладка:**
- Desktop/широкий: две колонки — слева слоты 1–10 (крупные), справа пул сеткой.
- Мобильный (`max-width: 640px`): боковая — слева узкая колонка 10 компактных слотов (номер + код/полоса команды), справа прокручиваемый пул компактными чипами (код + полоса цветом команды). Тап работает между колонками, обе видны одновременно, без вертикальной простыни.

**Read-only (locked, `now() > deadline_utc`):** слоты показывают сохранённый прогноз без интеракции; пул скрыт; вместо «Сохранить» — плашка «Дедлайн прошёл, прогноз зафиксирован» или «Ты не делал прогноз на эту гонку».

## 7. Функция `open_race()` и dev-бутстрап

**`open_race(p_race_id bigint) returns int`** (миграция 0007, `SECURITY DEFINER`, `search_path=public`):
1. Права: если не `is_admin()` и не service_role → `raise exception`.
2. Снимок пула: `insert into race_driver_pool (race_id, driver_id) select p_race_id, id from drivers where active on conflict do nothing`.
3. Статус: `update races set status='open' where id=p_race_id and status='demo'` (не трогает `closed`/`resulted` — защита от переоткрытия).
4. Возврат: число пилотов в пуле гонки.

Идемпотентна: повторный вызов не плодит дублей и не ломает уже открытую гонку. Переиспользуется админкой (2c) и автоматикой (Фаза 4). `grant execute ... to authenticated` (внутри гейтит по is_admin).

**Dev-бутстрап** `scripts/dev/bootstrap-open-belgium.js`: cloud-direct через `SUPABASE_DB_URL` (service-role обходит RLS), находит `id` гонки `season=2026, round=10`, зовёт `select open_race(id)`, печатает размер пула. Разовый.

## 8. Обработка ошибок (`db.ts`)

Маппинг серверных ошибок в сообщения:

| Серверная ошибка | Код | Сообщение UI |
|---|---|---|
| RLS отклонил insert/update (дедлайн) | `deadline` | «Дедлайн прошёл — прогноз больше нельзя изменить» |
| `exactly 10 drivers` / `10 distinct drivers` | `shape` | «Нужно заполнить все 10 мест разными пилотами» |
| `all drivers must be in the race pool` | `pool` | «Пилот не из состава этой гонки (обнови страницу)» |
| прочее | `unknown` | «Не удалось сохранить, попробуй ещё раз» |

RLS-отказ на записи Supabase возвращает как ошибку (нет строк / 42501-подобное) — `savePrediction` детектит и маппит в `deadline`, предварительно проверив `now() > deadline_utc` на клиенте для точности сообщения.

## 9. Тестирование

- `scripts/db/open_race.test.js` (pg, транзакция + rollback): снимок берёт только `active`; идемпотентность (2× вызов = тот же пул); не переоткрывает `resulted`; не-админ (SET ROLE authenticated без is_admin) получает отказ.
- Scoring/RLS/validation уже покрыты Фазой 0 — не дублируем.
- Фронт: ручной e2e-смоук в браузере (как в 2a): бутстрап открывает Бельгию → логин → разложить топ-10 tap-to-assign → сохранить → перезайти (видно сохранённое, редактируемо) → проверить мобильную боковую раскладку через DevTools. React-юнит-тестов в 2b нет (нет раннера; YAGNI до 2c).
