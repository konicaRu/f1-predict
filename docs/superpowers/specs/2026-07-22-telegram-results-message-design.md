# Расширенное сообщение об итогах гонки в Telegram

Дата: 2026-07-22
Статус: утверждено

## Зачем

Сейчас сообщение об итогах гонки (`scripts/telegram/notify.js`, режим `results`) шлёт только
факт-топ-10 и голый список «игрок — очки». Не видно ни самих прогнозов, ни того, кто выиграл этот
тур, ни общего зачёта сезона — за этим приходится идти на сайт. Хочется, чтобы в самом чате сразу
было видно и «кто как предсказал», и кто молодец в этом туре, и текущее положение в сезоне.

## Что не меняется

- Расписание проверки (`cron: '0 */2 * * *'`) — не спам: сообщение уходит один раз на гонку
  (`telegram_announced_at`), просто как часто проверяем, появился ли новый результат. В среднем
  задержка от финиша гонки до сообщения — около часа, максимум ~2 часа. Оставляем как есть.
- Режимы `raceweek`/`deadline`/`remind` — не трогаем.
- Идемпотентность (`telegram_announced_at is null` в выборке гонок) — без изменений.

## Что меняется

Только функция `results()` в `scripts/telegram/notify.js`. Новый формат сообщения:

```
🏁 Финиш Belgian Grand Prix!

Топ-10: 1.ANT 2.LEC 3.VER 4.HAM 5.PIA 6.HAD 7.NOR 8.BOR 9.LIN 10.COL

Прогнозы и очки:
1. Dim — подиум ANT-HAM-PIA → 69 (1 точное)
2. Iceman — подиум RUS-HAM-ANT → 43 (0 точных)

🏆 Лучший прогноз тура — Dim (69 очков)!

Общий зачёт сезона:
1. Dim — 69
2. Iceman — 43
3. Павел — 0
```

### Прогноз-подиум каждого игрока

Новый запрос к `predictions` по этой гонке (`race_id`, `user_id`, `positions`), первые 3 элемента
`positions` → коды через уже существующий `driverCodeMap()`/`codeFor()`. Только полный подиум
(P1-P3), не все 10 мест — иначе сообщение с несколькими игроками разрастается и плохо читается на
телефоне (обсуждали, осознанный компромисс).

Строка на игрока = существующая строка очков (`{n}. {имя} — {points} ({exact} точных)`), дополненная
подиумом прогноза перед стрелкой: `{n}. {имя} — подиум {P1}-{P2}-{P3} → {points} ({exact} точных)`.
Список игроков и их порядок — как сейчас, по убыванию очков (тот же `scoreRows`, из которого уже
берутся очки).

### Лучший прогноз тура

Игрок(и) с максимальным количеством очков за эту гонку (верх уже отсортированного `scoreRows`).
Если несколько игроков набрали одинаковый максимум — перечисляются через запятую в одной фразе:
`🏆 Лучший прогноз тура — {Имя1}, {Имя2} ({N} очков)!`. Тон фразы — прямой, без вычурности.

### Общий зачёт сезона

Новый SQL-запрос: сумма очков по каждому члену лиги только по гонкам с `races.scored = true`
(так же, как трактует «зачётность» фронтенд — не все `resulted`-гонки идут в зачёт, только
`scored=true`). Показываются все игроки лиги, не топ-N.

Тайбрейкер и ранжирование — портируем логику `aggregateStandings` из `src/lib/standings.ts` (та же,
что использует экран «Зачёт» на сайте, чтобы цифры в Telegram и на сайте не расходились):
1. Сумма очков по зачётным гонкам ↓
2. Сумма точных попаданий ↓
3. Лучший результат за одну гонку ↓
4. Имя (для стабильности при полном совпадении)

Ранг — соревновательный (равным по всем ключам — одно место, следующий сдвигается: 1,2,2,4), не
плотный. Игроки без единого прогноза в зачётных гонках тоже показываются, с 0 очков.

SQL для агрегата (фильтрация по зачётным гонкам через подзапрос, а не условие в `ON`, чтобы не
протащить очки за незачётные гонки через `LEFT JOIN`):

```sql
select u.id, u.display_name,
       coalesce(sum(cs.points), 0) as points,
       coalesce(sum(cs.exact_hits), 0) as exact,
       coalesce(max(cs.points), 0) as best_race
from users u
left join (
  select s.user_id, s.race_id, s.points, s.exact_hits
  from scores s
  join races r on r.id = s.race_id
  where r.scored = true
) cs on cs.user_id = u.id
group by u.id, u.display_name
```

Сортировка и присвоение соревновательного ранга (1,2,2,4) — в JS после запроса, тем же способом,
что и `aggregateStandings` (сравнение по четырём ключам подряд, ранг = позиция первого элемента
текущей группы совпадений).

## Вне охвата

- Не меняем частоту cron.
- Не трогаем `raceweek`/`deadline`/`remind`.
- Не выносим общий зачёт в отдельное сообщение — всё в одном.
- Не показываем полный прогноз (все 10 мест) — только подиум P1-P3.

## Тестирование

Юнит-тест на портированную логику ранжирования (та же, что уже покрыта тестами на фронте —
здесь не дублируем фронтовые тесты, а проверяем именно SQL→JS агрегацию: сумма очков, тайбрейкер,
соревновательный ранг на выдуманных данных). Плюс ручная проверка на реальных данных: Бельгия
(round 10) уже имеет `telegram_announced_at` — придётся сбросить его на тестовом прогоне вручную
(через `scripts/db`) либо просто прогнать функцию напрямую с выводом в консоль (`--dry-run` не
нужен — `sendTelegram` можно на время подменить на `console.log` для локальной проверки, либо
прогнать по-настоящему в тестовый чат/просто в боевой, раз это не создаёт дублей благодаря
`telegram_announced_at`).
