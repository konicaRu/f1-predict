# Telegram-напоминания и итоги гонок (дизайн)

Дата: 2026-07-19
Статус: согласован, готов к плану
Сверено с `docs/constitution.md` (§4 инфраструктура — с одним осознанным отступлением, см. §3 ниже;
§5 процесс; §6 YAGNI).

## 1. Цель

Автоматические сообщения в общий Telegram-чат лиги:
- **Гоночная неделя** (понедельник утром) — анонс, что на очереди Гран-при, дедлайн прогнозов — четверг.
- **Напоминания о дедлайне** (среда и четверг, дважды в день) — не забыть поставить прогноз.
- **Итоги гонки** (как только результат занесён финально) — топ-10 факт + очки игроков за эту гонку.

Реализует «Требование (принято 2026-06-30): итоги гонки в Telegram-чат» и мобильный/групповой сценарий
напоминаний из `docs/plan.md` §10, но в упрощённом (без тегов отстающих) и технически другом (GitHub
Actions вместо pg_cron) виде — см. §3.

## 2. Контекст (что уже готово)

- `races(id, round, name, deadline_utc, race_datetime_utc, status, scored)`, `results(race_id, positions
  jsonb, status, fetched_at)`, `scores` (view: user_id, race_id, points, exact_hits), `users(id,
  display_name)`, `drivers(id, code, name)`.
- Дедлайн — четверг, время фиксировано по МСК (хранится в UTC, `deadline_utc`).
- Cloud-direct паттерн уже отработан дважды: `scripts/import/`, `scripts/db/`, `scripts/export/` — прямое
  подключение к Postgres через `pg` и `SUPABASE_DB_URL` из `.env`/секретов, свой `package.json` на папку.
- GitHub Actions уже дважды доказали себя рабочими в этом проекте: `.github/workflows/deploy.yml` (push→
  build→Pages) и `.github/workflows/keepalive.yml` (расписание `schedule:`, секреты через `${{ secrets.* }}`).
- `docs/plan.md` §10 содержит более ранний, более сложный набросок (с тегами отстающих через onboarding
  `/start`) — сознательно НЕ реализуем эту часть сейчас (см. §7 вне скоупа).

## 3. Решения (из брейншторма)

- **Инфраструктура — GitHub Actions, не pg_cron/Edge Function.** Конституция §4 декларирует pg_cron как
  предпочтительный таймер, но ни pg_cron, ни Supabase Edge Functions ни разу не были использованы или
  проверены в этом проекте — локальный Docker-стек Supabase не работает на этой машине (сетевой блок,
  причина cloud-direct пивота ещё в Фазе 0), и `supabase functions deploy` никогда не тестировался с этой
  машины. Точность до минуты не критична для дружеских напоминаний (в отличие от keep-alive, где точность
  была нужна против таймера паузы Supabase) — GitHub Actions уже дважды доказал надёжность в этом самом
  проекте. **Это официальное расширение уже существующего исключения** (keep-alive точно так же не на
  pg_cron) — конституция обновляется этим же под-проектом (см. план, задача с правкой `docs/constitution.md`).
- **Без тегов отстающих участников.** Только общие сообщения в чат, без адресных тегов — не нужен
  onboarding `/start`/привязка Telegram-аккаунтов. Проще, соответствует тому, что фактически нужно сейчас.
- **Расписание (МСК, без DST в России — фиксированный сдвиг UTC+3):**
  - Понедельник 10:00 — `raceweek`
  - Среда 10:00 и 19:00, четверг 10:00 и 19:00 — `deadline` (один и тот же текст на все 4 слота)
  - Каждые 2 часа, круглосуточно — `results` (проверка новых финальных результатов)
- **Итоги гонки — только один раз на гонку.** Правка уже занесённого результата (override через Admin)
  НЕ должна триггерить повторное сообщение — нужен трекинг «уже объявлено» на уровне БД (новая колонка).
- **Содержание итогов — только результат этой гонки**, без снимка общего зачёта сезона (тот всегда
  доступен на сайте, вкладка «Зачёт»).
- **Формат сообщений — с эмодзи** (казуальный тон дружеской лиги, см. §6 для точных текстов).

## 4. Архитектура и файлы

```
scripts/telegram/
  package.json        — зависимости: pg (fetch — встроен в Node ≥18, доп. пакет не нужен)
  lib.js               — pg-подключение (по образцу scripts/export/lib.js) + sendTelegram(text) helper
  notify.js            — точка входа: node notify.js <raceweek|deadline|results>
  README.md            — инструкция по созданию Telegram-бота, добавлению в чат, получению chat_id
.github/workflows/
  telegram-notify.yml  — новый workflow, 6 cron-триггеров (см. §3), маппинг schedule→режим, секреты
supabase/migrations/
  0011_telegram_announced.sql — alter table races add column telegram_announced_at timestamptz
.gitignore            — без изменений (секреты бота — только в GitHub Actions secrets, локальный .env
                        не обязателен, если не тестируешь `results`/`deadline` локально; если тестируешь —
                        та же схема, что для Google-ключа: переменные в .env, не в git)
```

**Границы:** `lib.js` — только подключение к БД + отправка одного текстового сообщения в Telegram (`fetch`
на `https://api.telegram.org/bot<TOKEN>/sendMessage`). `notify.js` — вся логика трёх режимов (SQL-запрос
+ форматирование текста + вызов `sendTelegram`), выбор режима — первый аргумент командной строки
(`process.argv[2]`).

## 5. Логика трёх режимов

Общий приём для «эта неделя»: `date_trunc('week', <момент> at time zone 'Europe/Moscow') = date_trunc('week', now() at time zone 'Europe/Moscow')`
— ISO-неделя (понедельник-старт), корректно ловит текущую МСК-неделю без ручной арифметики по дням.

- **`raceweek`** (запуск по понедельникам): найти `races` со `status='open'`, чей `deadline_utc` попадает
  в текущую МСК-неделю. Если нет ни одной — ничего не отправлять (пауза в календаре, например межсезонье
  или неделя без Гран-при). Если есть — отправить `raceweek`-сообщение с именем/раундом гонки.
- **`deadline`** (запуск ср/чт): та же выборка, что и `raceweek` (`status='open'`, дедлайн на этой неделе),
  плюс условие `deadline_utc > now()` (если дедлайн уже прошёл к моменту запуска — не шлём, гонка уже
  закрыта для прогнозов). Текст один и тот же на все 4 временных слота.
- **`results`** (запуск раз в 2 часа): `races` со `status='resulted' AND scored=true AND
  telegram_announced_at IS NULL`. Для каждой найденной (обычно 0 или 1, но код не полагается на «ровно
  одна») — собрать топ-10 (join `results.positions` → `drivers.code`) и очки игроков (`scores` view join
  `users.display_name`, сортировка по очкам ↓), отправить сообщение, затем `UPDATE races SET
  telegram_announced_at = now() WHERE id = $1` — это и есть защита от повторной отправки при override.

## 6. Тексты сообщений (Telegram `parse_mode: HTML`, простой `<b>` для акцентов)

- **`raceweek`:**
  ```
  🏎 Гоночная неделя! На очереди <b>{name}</b> (раунд {round}).
  Дедлайн прогнозов — четверг {ЧЧ:ММ} МСК.
  Ставь: {SITE_URL}/predict
  ```
- **`deadline`:**
  ```
  ⏰ Не забудь поставить прогноз на <b>{name}</b>!
  Дедлайн — четверг {ЧЧ:ММ} МСК.
  {SITE_URL}/predict
  ```
- **`results`:**
  ```
  🏁 Финиш <b>{name}</b>!

  Топ-10:
  1. {code} 2. {code} 3. {code} 4. {code} 5. {code}
  6. {code} 7. {code} 8. {code} 9. {code} 10. {code}

  Очки за гонку:
  1. {display_name} — {points} ({exact_hits} точных)
  2. ...
  ```
  Время дедлайна и `SITE_URL` (`https://konicaru.github.io/f1-predict`) — константы в `notify.js`, время
  форматируется из `deadline_utc` через `toLocaleString('ru-RU', {timeZone:'Europe/Moscow', hour:'2-digit',
  minute:'2-digit'})` (тот же приём, что уже использован в `scripts/export/export.js`'s `toMsk`).

## 7. Секреты и ручная настройка (GitHub Actions repo secrets)

- `TELEGRAM_BOT_TOKEN` — токен бота от @BotFather.
- `TELEGRAM_CHAT_ID` — id общего группового чата (получается через `getUpdates` после того, как бот
  добавлен в группу и там написано любое сообщение).
- `SUPABASE_DB_URL` — тот же прямой коннекшн, что уже в локальном `.env` для `scripts/export`/`scripts/import`;
  добавляется как **отдельный** GitHub secret (тот же уровень доверия, что и локальный `.env` — просто то
  же самое хранится ещё и в GitHub Actions secrets, а не только на диске разработчика).
- **Ручная часть (вне доступа агента):** создание бота, добавление в чат, получение `chat_id` — пошаговая
  инструкция в `scripts/telegram/README.md`, по образцу `scripts/export/README.md`.

## 8. Тестирование

- Ручной прогон каждого режима локально против реальной БД: `node scripts/telegram/notify.js raceweek`,
  `deadline`, `results` — с уже настроенными `.env`-переменными (`TELEGRAM_BOT_TOKEN`,
  `TELEGRAM_CHAT_ID`, `SUPABASE_DB_URL`), проверка сообщения в чате глазами.
  - `raceweek`/`deadline` тестируются на реально открытой гонке (сейчас это Венгрия/Нидерланды).
  - `results` тестируется по уже отработанной в проекте схеме фикстур: временно занести результат на
    гонку-подопытную (round вне активной игры), прогнать `results`-режим, проверить сообщение и что
    `telegram_announced_at` проставился, затем **полностью откатить** (удалить `results`, обнулить
    `telegram_announced_at`, вернуть `races.status='demo', scored=false`) — та же дисциплина очистки, что
    в Фазах 2c/3. Бельгию (round 10, играется прямо сейчас) не трогать.
- `.github/workflows/telegram-notify.yml` проверяется через `workflow_dispatch` (ручной запуск из вкладки
  Actions с явным указанием режима) — так же, как уже делалось для `deploy.yml`/`keepalive.yml`.
- Автотестов на сам вызов Telegram Bot API нет (внешний сервис) — SQL-часть логики можно проверить
  отдельно через `scripts/db/runner.js`, как уже делалось в Task 2 плана бэкапа в Sheets.

## 9. Вне скоупа

- Теги отстающих участников (`@username`) — требует onboarding `/start` и таблицы telegram_id↔user_id.
  Отдельная фича, если понадобится позже.
- Снимок общего зачёта сезона в сообщении об итогах.
- Provisional-результаты (T+4ч) — сообщение шлётся только на `final`.
- Обновление ранее отправленного сообщения при исправлении результата («результат уточнён») — сейчас
  просто не шлём повторно; отдельное сообщение-уточнение можно добавить позже при реальной необходимости.
- Приём прогнозов прямо в Telegram-боте (отдельная идея из беклога).
