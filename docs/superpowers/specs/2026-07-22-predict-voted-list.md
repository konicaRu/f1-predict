# Список проголосовавших на вкладке «Прогноз»

Дата: 2026-07-22
Статус: утверждено

## Зачем

Игроки просили видеть на вкладке «Прогноз», кто из лиги уже сделал прогноз на текущую гонку —
как стимул не откладывать и лёгкий соревновательный элемент. Показываем только факт «сделал
прогноз», без раскрытия содержимого — сами прогнозы, как и раньше, видны только после дедлайна
(конституция §1, RLS не меняется).

## Технический контекст

RLS на `predictions` (`supabase/migrations/0003_rls.sql`, политика `pred_select_own` +
`pred_select_after_deadline`) скрывает **саму строку** чужого прогноза до дедлайна — не только
`positions`, но и факт существования строки. Обычный `select user_id from predictions where
race_id = X` с фронта до дедлайна вернёт только свою собственную строку. Чтобы показать список
проголосовавших до дедлайна, нужна отдельная server-side функция, которая отдаёт **только
`user_id`** (не `positions`) в обход построчного RLS — по прецеденту `keepalive_ping()`
(`supabase/migrations/0008_keepalive.sql`, `security definer`).

## Что меняется

### Бэкенд

Новая миграция `supabase/migrations/0012_predicted_user_ids.sql`:

```sql
-- Отдаёт только user_id тех, кто сделал прогноз на гонку — НЕ positions.
-- security definer обходит RLS-скрытие чужих строк до дедлайна (сознательно, узко:
-- наружу выходит только факт участия, содержимое прогноза остаётся скрытым).
create or replace function public.predicted_user_ids(p_race_id bigint)
returns setof uuid
language sql security definer set search_path = public as $$
  select user_id from public.predictions where race_id = p_race_id;
$$;

grant execute on function public.predicted_user_ids(bigint) to authenticated;
```

### Фронтенд

`src/lib/db.ts` — новая функция:

```ts
export async function getVotedUserIds(raceId: number): Promise<string[]> {
  return withRetry(async () => {
    const { data, error } = await supabase.rpc('predicted_user_ids', { p_race_id: raceId });
    if (error) throw error;
    return (data ?? []) as string[];
  });
}
```

`src/pages/Predict.tsx`:
- Дополнительно загружает `listUsers()` (уже есть, используется в `Results.tsx`) и
  `getVotedUserIds(Number(raceId))` вместе с остальными данными гонки.
- В шапке страницы (`predict-head`), под строкой таймера/дедлайна, рендерится строка:
  ```
  ✓ Поставили: Dim, Iceman
  ```
  Имена — из `listUsers()`, отфильтрованные по множеству `votedUserIds`, отсортированные по
  алфавиту (тот же `localeCompare`, что в `notVotedNames` в `scripts/telegram/notify.js`). Тега
  «ты» нет — просто общий список, единый для всех.
- Если `votedUserIds` пуст (никто ещё не сделал прогноз) — строка не рендерится вообще.
- После успешного `save()` список перезапрашивается (тот же вызов `getVotedUserIds`), чтобы своё
  имя появилось в строке сразу, без перезагрузки страницы.
- Работает одинаково до и после дедлайна — функция не завязана на дедлайн, отдельного пути для
  `readOnly`-режима не нужно.

## Вне охвата

- Не показываем, кто НЕ сделал прогноз (это уже есть в Telegram-напоминании по четвергам, здесь —
  сознательно только позитивный список).
- Не показываем сами прогнозы/составы до дедлайна — RLS на `positions` не меняется, функция
  отдаёт только `user_id`.
- Не делаем realtime-обновление (подписку на изменения) — список обновляется при загрузке
  страницы и после своего сохранения, этого достаточно для цели «стимул успеть».

## Тестирование

- Ручная проверка в браузере (`npm run dev`): открыть «Прогноз» под двумя разными аккаунтами
  (или одним админским + прямой SQL-вставкой тестового прогноза второго игрока), убедиться что
  до дедлайна виден только факт «поставил», не сам прогноз.
- Прогнать миграцию на реальной БД, вызвать `predicted_user_ids` напрямую SQL-запросом и
  сверить, что возвращает только `user_id`, без `positions`.
- Проверить, что после своего сохранения имя появляется в строке без перезагрузки страницы.
- `npx tsc --noEmit` должен проходить чисто.
