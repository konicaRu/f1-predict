# AI-игрок в лиге (GridBot) — Design

**Статус:** дизайн согласован с пользователем (брейншторм 2026-07-24), готов к переносу в implementation plan.

## 1. Идея и мотивация

Бэклог-пункт `docs/plan.md` §15 «ИИ-игрок в лиге» (записан 2026-07-16, без реализации): подключить
бесплатный ИИ-API — модель делает прогноз на топ-10 наравне с людьми, играет в общем зачёте.

**Цель — фан-элемент, не «серьёзный конкурент».** Прогноз должен быть правдоподобным (не наугад), но
не требуется бороться за максимальную точность любой ценой. Бюджет — только бесплатный тариф.

## 2. Кто это: GridBot как обычный участник

GridBot — настоящая строка в `auth.users`/`public.users` (`display_name='GridBot'`, `is_admin=false`),
не отдельная сущность и не отдельный UI. Играет по тем же правилам: та же формула очков
(`score_prediction`), тот же общий зачёт (`scores`), появляется в списке «Проголосовали» на вкладке
«Прогноз» как обычный игрок. Никаких изменений во фронтенде не требуется.

**Риск, который нужно проверить эмпирически при реализации (не считать решённым заранее):** прямая
вставка в `auth.users` в обход штатного флоу Supabase Auth (signup) — прецедента в проекте нет (тестовые
файлы вставляют туда синтетические строки, но внутри отката транзакции `rollback`, это никогда не
проверялось как ПОСТОЯННАЯ запись). Перед тем как писать окончательную миграцию, нужно тем же
приёмом (`begin; ...; rollback;` через `scripts/db/runner.js sql`), что и в security-фиксе 2026-07-22,
подтвердить, какие колонки `auth.users` обязательны и что после вставки провал не ломает ничего в
Supabase Auth (напр. панель Auth Users в дэшборде, будущие обычные signup'ы).

## 3. Данные — источники и объём

Три источника, по возрастанию сложности:

1. **Пул этой гонки** — `race_driver_pool` + `drivers` (код, имя, команда). Уже есть.
2. **Текущий сезон** — свои прошедшие результаты `results`/`races` (топ-10 по каждому прошедшему
   раунду 2026) + `drivers.standing` (текущее место в чемпионате). Уже есть в БД.
3. **Прошлые сезоны F1** — **живой запрос к Jolpica** (тот же API, что `scripts/import`, тот же паттерн
   `fetchJolpica` из `scripts/import/lib.js`: `https://api.jolpi.ca/ergast/f1/{season}/driverStandings.json`).
   Берём итоговое положение (очки, место) каждого гонщика пула за последние 3 сезона (2023–2025) —
   компактный сигнал «карьерная траектория», не построчная история каждой гонки (это было бы слишком
   много данных и API-вызовов ради фан-фичи).

**Явно вне скоупа (решено в брейншторме):** отдельный источник по Формуле-2/младшим сериям для
новичков. Вместо интеграции со вторым API — модели просто говорим в промпте: «этот гонщик новичок в
F1, структурированных данных о его прошлом нет, используй свои общие знания, если они есть». Осознанный
компромисс ради фан-фичи, не серьёзного предиктора.

## 4. Промпт и модель

- **Провайдер:** Google Gemini API (генеративный REST, ключ через aistudio.google.com — у пользователя
  уже есть аккаунт). Бесплатный тариф с запасом хватает при частоте «раз в неделю».
- **Интеграция — сырой `fetch` к `generativelanguage.googleapis.com`**, без нового npm-пакета
  (Node 20+ имеет нативный `fetch`) — по тому же минимализму, что `scripts/telegram/lib.js` (сырой
  `fetch` к Telegram Bot API, без SDK).
- **Модель — через env `GEMINI_MODEL`, дефолт `gemini-2.0-flash`** (легко сменить без правки кода).
- **Структурированный JSON-режим** (`responseMimeType: 'application/json'` + `responseSchema`) — модель
  обязана вернуть `{ "top10": [10 кодов гонщиков], "reasoning": "1-2 предложения" }`. Никакого
  парсинга свободного текста.
- Промпт включает: список пула (код/имя/команда), собственные результаты этого сезона по раундам,
  итоги последних 3 сезонов по каждому гонщику пула (или пометку «новичок, данных нет» для тех, у
  кого нет истории), явную инструкцию про топ-10 из ровно данного пула.

## 5. Валидация и подстраховка (fallback)

Перед сохранением — собственная проверка (не полагаемся только на серверный триггер
`validate_prediction`): ровно 10 разных кодов, все из пула этой гонки.

Если ответ модели невалиден, API недоступен, таймаут или лимит — **не роняем прогон**: подставляем
эвристику — сортировка пула по `drivers.standing` (без места — в конец), берём первые 10. GridBot
всё равно поставит прогноз (пусть не «умный» в этот конкретный раз), плюс `console.warn` с причиной.
Симметрично паттерну `try/catch` на каждую гонку в `scripts/autoresults/fetch.js` — единичный сбой не
должен ломать остальную автоматику.

## 6. Сохранение прогноза

Прямая запись в `predictions` через прямое подключение к Postgres (`SUPABASE_DB_URL`, тот же
cloud-direct паттерн, что `scripts/autoresults`/`scripts/import` — в обход RLS, как и остальная
серверная автоматика проекта): `insert ... on conflict (user_id, race_id) do update`. Ищем гонку тем
же способом, что и `thisWeekOpenRaces()` в `scripts/telegram/notify.js` (status='open', дедлайн в этой
неделе) — переиспользовать хелпер или скопировать запрос.

## 7. Telegram-объяснение

**Только по четвергам** (день дедлайна) — но так как сам скрипт запускается по крону только по
четвергам (см. §8), внутри скрипта отдельная проверка дня недели не нужна, в отличие от
`notify.js:deadline()`, который гоняется в несколько дней и сам вычисляет `isMskThursday()`.

После успешного сохранения прогноза — короткое сообщение в общий чат через уже существующий
`sendTelegram()` из `scripts/telegram/lib.js`:

```
🤖 GridBot поставил на <Название Гран-при>: VER-NOR-PIA-...
Почему: <reasoning из ответа модели>
```

Если сработал fallback (не модель) — объяснение не постим (нечего объяснять) или короткое
«не получилось спросить ИИ, поставил по текущему чемпionату» — решить в плане.

## 8. Запуск и расписание

Новый режим `aiplayer` в `scripts/ai-player/predict.js`, встроенный в **уже существующий**
`.github/workflows/telegram-notify.yml` (не отдельный workflow) — расширяем список режимов на
существующем четверговом триггере `'5 7 * * 4'` (10:05 МСК): было `deadline`, станет `aiplayer deadline`
(сначала прогноз, потом напоминание — по тому же принципу, что понедельничный `raceweek remind`).
10:05 МСК заведомо раньше вечернего дедлайна (пример — Hungarian GP дедлайн был 20:00 МСК того же дня).

## 9. Секреты / ручная настройка

- `GEMINI_API_KEY` — новая переменная в `.env`/`.env.example` + GitHub-секрет в workflow. Ручной шаг:
  сгенерировать ключ на aistudio.google.com (аккаунт уже есть) — проведу по шагам, как с ботом/Sheets.
- Миграция `supabase/migrations/00XX_gridbot_user.sql` — создание аккаунта GridBot (см. риск в §2).

## 10. Тестирование

По установленной в проекте конвенции:
- Чистая логика (валидация топ-10, fallback-сортировка, сборка промпта/парсинг ответа) —
  plain console-harness тест `scripts/ai-player/predict.test.js`, по образцу `notify.test.js`.
- Части, трогающие БД (вставка аккаунта GridBot, сохранение прогноза) — тест в откатываемой
  транзакции (`begin; ...; rollback;`), по образцу `predicted_user_ids.test.js`/`security_grants.test.js`.
- Реальный вызов Gemini API — не юнит-тестируется дёшево; финальная проверка — живой смок-прогон на
  реальной открытой гонке (по аналогии с тем, как autoresults/telegram проверялись на боевой Бельгии),
  с ручным подтверждением пользователем прогноза + сообщения в Telegram.

## 11. Явно вне скоупа (v1)

- Коэффициенты букмекеров как источник данных — отклонено в брейншторме (лишняя интеграция).
- Отдельный источник по Формуле-2/младшим сериям — отклонено, полагаемся на знания модели.
- UI-изменения — не нужны, GridBot неотличим от обычного игрока в интерфейсе.
