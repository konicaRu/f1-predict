## 🎮 Что это и как этим пользоваться

Каждый участник заходит на сайт, открывает вкладку **«Прогноз»** и до дедлайна (четверг,
фиксированное время по МСК, см. страницу гонки) drag-and-drop'ом расставляет 10 карточек пилотов
по местам P1–P10. Прогноз можно менять сколько угодно раз до дедлайна — сохраняется последняя
версия.

Жизненный цикл гонки в системе (поле `races.status`):

```
demo → open → resulted
 │      │        │
 │      │        └─ результат внесён (авто или вручную), очки посчитаны, зачёт обновлён
 │      └─ можно ставить прогноз, дедлайн ещё не наступил / только наступил
 └─ архивные гонки до старта лиги — витрина без начисления очков
```

Прогнозы других участников **скрыты, пока не пройдёт дедлайн этой гонки** — это не UI-заглушка
(типа "спрятать блок"), а политика доступа к строкам таблицы прямо в базе данных (подробнее —
[RLS](#-схема-данных-и-rls) ниже). Даже если кто-то откроет консоль разработчика в браузере
и напрямую дёрнет API Supabase, база сама не отдаст чужие прогнозы раньше времени.

После дедлайна на вкладке «Прогноз» видно, кто уже проголосовал (без самих прогнозов — они
раскрываются только после гонки). После гонки результат заносится в систему — либо
автоматически (см. [автозабор результатов](#-telegram-уведомления)), либо администратор лиги
вносит его вручную на вкладке «Админка». Как только результат финальный, для каждого участника
считаются очки, обновляется общий зачёт сезона и в Telegram уходит сообщение с топ-10, разбором
прогнозов и таблицей общего зачёта.

## 🧮 Формула очков

Идея: чем точнее и увереннее прогноз (позиция ближе к началу топ-10), тем больше очков можно
получить и тем больнее промах.

```
points(слот i) = max(0, WEIGHTS[i] − 2·|реальная_позиция − заявленная_позиция|) + (3, если попал точно)

WEIGHTS = [25, 18, 15, 12, 10, 8, 6, 4, 2, 1]   // вес за позиции P1..P10 соответственно
```

Если пилот, которого поставили в топ-10, в реальной гонке финишировал ниже 10-го места (или
сошёл) — за этот слот 0 очков, штрафа "в минус" не бывает (`max(0, ...)`).

**Пример.** Игрок поставил VER на P1. VER реально финишировал третьим.
- `i = 0` → `WEIGHTS[0] = 25`, `Y = 1` (заявленная позиция), `X = 3` (реальная)
- `points = max(0, 25 − 2·|3 − 1|) = max(0, 25 − 4) = 21` очко за этот слот (бонус +3 не
  начисляется, так как позиция не совпала точно)

Если бы VER действительно финишировал первым (`X = Y = 1`): `points = 25 − 0 + 3 = 28`.

Тайбрейкер общего зачёта при равенстве очков: больше точных попаданий → выше место; если и это
равно — выше результат в лучшей отдельной гонке.

Реализация на фронтенде — [`src/lib/scoring.ts`](src/lib/scoring.ts); на бэкенде та же формула
продублирована SQL-функцией `score_prediction` в
[`supabase/migrations/0002_scoring.sql`](supabase/migrations/0002_scoring.sql), которая и
является источником истины для реальных очков (view `scores`) — фронтенд использует свою копию
только для мгновенного превью во время расстановки прогноза.

## 🧠 Почему так устроено (архитектурные решения)

Ниже — не просто список технологий, а объяснение, зачем каждая была выбрана именно такой. Это
полезно прочитать перед тем, как менять что-то в проекте.

**Нет своего бэкенд-сервера.** Вместо Node/Express с собственной бизнес-логикой на сервере —
[Supabase](https://supabase.com) (управляемый Postgres + Auth + Edge Functions). Игроков мало
(это лига для компании друзей, не публичный продукт), нагрузка нулевая, а держать и обновлять
отдельный сервер под неё — чистые накладные расходы. Фронтенд (статика на GitHub Pages) обращается
к Supabase напрямую по HTTPS с публичным `anon`-ключом.

**Anon-ключ публичный — и это нормально.** Ключ, которым пользуется браузер, лежит прямо в
собранном JS-бандле — его при желании увидит кто угодно. Поэтому вся безопасность проекта
держится **не на секретности ключа**, а на том, что сама база данных под этим ключом отдаёт и
принимает только то, что разрешено — см. [RLS](#-схема-данных-и-rls). Это стандартный паттерн
для Supabase-проектов с публичным фронтендом.

**«Cloud-direct» вместо локального Docker.** Обычно для Supabase поднимают локальный Docker-стек
для разработки и тестов. На машине, где велась разработка, Docker не завёлся (сетевые
ограничения), поэтому все инструменты в `scripts/*/` подключаются **напрямую к облачной
Postgres** через connection string из `.env` (`SUPABASE_DB_URL`). Из этого следует важное
практическое правило: `npm run rebuild` в `scripts/db` **не тестовая песочница**, а прямой снос
и пересоздание всех таблиц в реальной облачной базе — команда, которую нельзя гонять не думая.

**GitHub Actions вместо pg_cron.** Расписания напоминаний, автозабора результатов и запуска
GridBot — это `.github/workflows/telegram-notify.yml` с cron-триггерами GitHub Actions, а не
`pg_cron` внутри Postgres (несмотря на то, что такая идея обсуждалась на старте проекта). GitHub
Actions оказался достаточно точным для нужд лиги (расписание сдвинуто на `:05`, а не `:00`, чтобы
не попадать в самую загруженную минуту очереди GitHub — см. комментарий в самом workflow-файле) и
не требует дополнительных расширений Postgres.

**Free-tier Supabase засыпает — нужен keep-alive.** Бесплатный план Supabase останавливает
проект после ~7 дней без активности к базе. `keepalive.yml` дважды в день делает настоящий
`select` к таблице (не просто пинг HTTP-эндпоинта — важно, чтобы запрос реально дошёл до
Postgres). Один инцидент уже был (2026-07-14): более редкое расписание (2 раза в неделю)
оказалось недостаточным, и проект уснул — с тех пор расписание 2×/день.

**Секреты — только в `.env` и GitHub Secrets, никогда в коде.** Локально все токены/ключи лежат
в корневом `.env` (в `.gitignore`, никогда не коммитится). Для автозапуска в GitHub Actions те же
значения продублированы в **Settings → Secrets and variables → Actions** репозитория — workflow
на лету собирает из них временный `.env` прямо во время запуска (см. шаг «Записать `.env`» в
`telegram-notify.yml`).

## 🗄️ Схема данных и RLS

### Таблицы

```
auth.users (управляется Supabase Auth)
     │ id
     ▼
public.users ──────────┐  display_name, is_admin, telegram_username
     │ id (user_id)     │
     ▼                  │
public.predictions      │   race_id, positions (jsonb: 10 driver_id по порядку P1..P10)
     │ race_id          │
     ▼                  │
public.races ───────────┤   season, round, deadline_utc, status, scored
     │ race_id          │
     ▼                  │
public.results          │   positions (jsonb: реальный топ-10), status (provisional/final)
                         │
public.race_driver_pool │   race_id + driver_id — какие пилоты доступны именно в этой гонке
     │ driver_id        │
     ▼                  │
public.drivers ─────────┘   code (VER, NOR...), name, team, team_color

public.result_changes       журнал правок результата: before/after/reason/changed_at
```

- `predictions.positions` и `results.positions` — это JSON-массив из 10 `driver_id`, индекс
  массива = позиция минус один (`positions[0]` — это P1). Уникальность прогноза на пару
  (игрок, гонка) обеспечена ограничением `unique (user_id, race_id)`.
- `race_driver_pool` существует отдельно от общего справочника `drivers`, потому что состав
  участников гонки может отличаться от сезона к сезону (замены, дебютанты) — это снимок «кто
  реально стартует» на конкретный раунд.
- Полная схема со всеми полями и комментариями — [`supabase/migrations/0001_schema.sql`](supabase/migrations/0001_schema.sql).

### RLS (Row Level Security) — как именно защищены данные

RLS — это встроенный в Postgres механизм: на таблицу можно навесить SQL-политики, и база **сама**
на уровне каждой строки решает, можно ли её вернуть/изменить конкретному пользователю — вне
зависимости от того, что просит клиент. Даже если фронтенд отправит "дай мне все прогнозы",
Postgres physически не отдаст лишнее.

В проекте это выглядит так (полностью — [`supabase/migrations/0003_rls.sql`](supabase/migrations/0003_rls.sql)):

| Таблица | Кто видит | Кто пишет |
|---|---|---|
| `predictions` | свою запись — всегда; чужую — только если `now() > races.deadline_utc` для этой гонки | только свою, только до дедлайна (после дедлайна `insert`/`update` политики просто не пропускают запрос); удаление вообще запрещено — политики `delete` нет |
| `users` | все аутентифицированные видят всех (нужно для списка участников/зачёта) | только свою строку, и только безопасные колонки (`display_name`, `telegram_username`, `telegram_user_id`) — колонка `is_admin` защищена отдельным `revoke`+точечным `grant`, обычным `update` её не подвинуть |
| `drivers` / `races` / `race_driver_pool` / `results` | все аутентифицированные читают всё | только админ (`is_admin() = true`) или сервисное подключение в обход RLS |
| анонимный (не залогиненный) пользователь | ничего | ничего |

Проверка «админ ли я» вынесена в отдельную SQL-функцию `is_admin()` с пометкой
`security definer` — это специально сделано, чтобы функция сама не попадала под RLS таблицы
`users` (иначе получилась бы рекурсия: чтобы проверить права, нужно прочитать `users`, а чтобы
прочитать `users`, нужно сначала пройти RLS, которая опять зовёт `is_admin()`...).

Дедлайн для записи проверяется **тем же SQL-условием прямо в политике** (`now() <= r.deadline_utc`),
а не в JS-коде фронтенда — значит, подделать системное время в браузере или обойти проверку через
прямой запрос к API бесполезно: решение принимает база, а не клиент.

Отдельная линия обороны — миграция [`0015_users_display_name_unique.sql`](supabase/migrations/0015_users_display_name_unique.sql):
`display_name` сделан уникальным, чтобы никто не мог переименовать себя в «GridBot» и таким
образом перехватить его прогноз (это реальная уязвимость, найденная на финальном ревью перед
слиянием GridBot в `main` — раньше `display_name` ничем не был защищён от дублей).

## 🤖 GridBot — участник-ИИ

В лиге играет ещё один «человек» — **GridBot**, обычный аккаунт в `public.users`
(миграция [`0014_gridbot_user.sql`](supabase/migrations/0014_gridbot_user.sql)), который получает
очки, участвует в общем зачёте и виден в списке проголосовавших наравне со всеми. Отдельного UI у
него нет — с точки зрения RLS и очков это такой же игрок, как и остальные.

### Откуда берётся прогноз

Раз в неделю (по четвергам, до дедлайна) запускается [`scripts/ai-player/predict.js`](scripts/ai-player/predict.js):

1. **Собирает контекст.** Берёт состав пилотов именно этой гонки (`race_driver_pool`), собственные
   результаты GridBot за уже прошедшие гонки текущего сезона, и через Jolpica API — итоговые
   места пилотов в личном зачёте за **три последних сезона** (2023–2025). Для пилотов-новичков
   без истории в Ф1 в промпте так и написано «новичок» — модели предлагается опереться на общие
   знания о его карьере в младших сериях.
2. **Строит текстовый промпт** (`buildPrompt` в `predict.js`) и отправляет в Gemini API в режиме
   структурированного JSON-ответа (`responseSchema` заставляет модель вернуть ровно объект вида
   `{ top10: [...10 кодов пилотов...], reasoning: "..." }` — не просто текст, а гарантированно
   парсящийся JSON).
3. **Проверяет ответ.** `isValidTop10()` требует: ровно 10 элементов, все уникальные, все входят
   в состав именно этой гонки. Если Gemini недоступна (сетевая ошибка, квота исчерпана) или ответ
   не прошёл проверку — в дело идёт **fallback**: `fallbackTop10()` просто сортирует пилотов по
   текущему месту в чемпионате. Так прогон никогда не падает полностью и GridBot гарантированно
   успевает проголосовать до дедлайна.
4. **Сохраняет прогноз** в `predictions` тем же способом, что и обычный игрок (`insert ... on
   conflict do update`) — от лица пользователя, которого находит по `display_name = 'GridBot'`.
5. Если Gemini ответила успешно (то есть есть `reasoning`) — публикует в общий Telegram-чат
   короткое объяснение вида:
   > 🤖 GridBot поставил на **Hungarian Grand Prix**: VER-NOR-PIA-...
   > Почему: <1-2 предложения от модели>

### Технические детали

- Модель — `gemini-2.5-flash` (бесплатный тариф Google AI Studio). Изначально был задан
  `gemini-2.0-flash`, но при первом реальном запуске оказалось, что у него нулевая бесплатная
  квота именно на этом API-ключе (HTTP 429, `limit: 0`) — модель заменили на `2.5-flash` по
  результатам списка доступных моделей через `ListModels`.
- Секрет — `GEMINI_API_KEY`, получить на [Google AI Studio](https://aistudio.google.com/) → **Get
  API key**. Нужен и локально (в `.env`), и как GitHub Secret репозитория (для автозапуска).
- Логика полностью изолирована в `scripts/ai-player/` — своя `package.json`, свой `lib.js` (общие
  хелперы: подключение к БД, запрос к Jolpica, запрос к Gemini, отправка в Telegram — специально
  продублированы, а не шарятся между `scripts/*`, ради независимости папок друг от друга).

### Запустить руками / проверить

```bash
cd scripts/ai-player
npm install
npm run predict
```

Если на этой неделе нет открытой гонки с дедлайном — скрипт так и напишет и ничего не сделает.
**Важно:** день недели не проверяется в самом скрипте — это чисто расписание крона в GitHub
Actions решает, что `aiplayer`-режим гоняется по четвергам. При ручном локальном запуске в любой
другой день, если гонка открыта и есть рабочий ключ Gemini, сообщение реально уйдёт в боевой
Telegram-чат — учитывайте это перед тем как тестировать вручную.

Полный сценарий разработки этой фичи — [`docs/superpowers/specs/2026-07-24-ai-player-design.md`](docs/superpowers/specs/2026-07-24-ai-player-design.md) и [`docs/superpowers/plans/2026-07-24-ai-player.md`](docs/superpowers/plans/2026-07-24-ai-player.md).

## 📣 Telegram-уведомления

Все сообщения уходят в **один общий групповой чат лиги** через обычного Telegram-бота (Bot API,
без веб-хуков — просто HTTP POST на `sendMessage`). Никакого отдельного сервера для бота не нужно:
всё гоняется по расписанию через GitHub Actions.

### Какие бывают сообщения

| Режим (`mode`) | Когда шлётся | Что внутри |
|---|---|---|
| `raceweek` | понедельник, если на этой неделе открыта гонка | 🏁 «На очереди Гран-при …», дедлайн, ссылка на «Прогноз» |
| `deadline` | среда и четверг (утром и вечером), если дедлайн ещё впереди | ⏰ напоминание поставить прогноз; по четвергам — ещё и список тех, кто **ещё не проголосовал** |
| `autoresults` | каждые 2 часа | не шлёт сообщений сам — молча пытается найти результат прошедшей гонки (Jolpica → OpenF1 как фолбэк) и занести его в базу |
| `results` | каждые 2 часа, сразу после `autoresults` | если появился новый финальный результат — 🏁 «Финиш …», топ-10, разбор прогнозов каждого игрока с очками, победитель тура, таблица общего зачёта сезона |
| `remind` | понедельник | ⚠️ если гонка уже прошла, а автозабор так и не нашёл результат — напоминание администратору занести вручную в Админке |
| `aiplayer` | четверг утром, до дедлайна | не шлёт сообщение сам по расписанию — запускает прогноз [GridBot](#-gridbot--участник-ии), который *может* сам отправить объяснение своего выбора |

Каждое сообщение о сайте (например, ссылка на «Прогноз» или «Админку») сопровождается пометкой
**«(нужен VPN)»** — GitHub Pages из РФ без VPN не открывается, добавлено отдельным фиксом после
жалобы, что человек переходил по ссылке и не понимал, почему сайт не грузится.

### Расписание (UTC → МСК)

GitHub Actions хранит расписание в UTC; Россия не переходит на летнее время, поэтому МСК = UTC+3
круглый год. Минуты специально сдвинуты на `:05` вместо `:00` — GitHub официально предупреждает,
что запуски по cron могут задерживаться именно в начале часа из-за наибольшей нагрузки.

| Cron (UTC) | Время в МСК | Что запускается |
|---|---|---|
| `5 7 * * 1` | Пн 10:05 | `raceweek` + `remind` |
| `5 7 * * 3` | Ср 10:05 | `deadline` |
| `5 16 * * 3` | Ср 19:05 | `deadline` |
| `5 7 * * 4` | Чт 10:05 | `aiplayer` + `deadline` (именно в этом порядке — GridBot успевает поставить прогноз до того, как людям придёт напоминание) |
| `5 16 * * 4` | Чт 19:05 | `deadline` |
| `5 */2 * * *` | каждые 2 часа | `autoresults` + `results` |

Вся логика выбора режима по расписанию — в шаге «Определить режимы» файла
[`.github/workflows/telegram-notify.yml`](.github/workflows/telegram-notify.yml).

### Разовая настройка бота (для тех, кто разворачивает лигу с нуля)

1. В Telegram написать [@BotFather](https://t.me/BotFather) → `/newbot` → задать имя и username
   (должен заканчиваться на `bot`). В ответ придёт токен вида `123456:ABC-DEF...` — это
   `TELEGRAM_BOT_TOKEN`.
2. Добавить бота в общий групповой чат лиги как обычного участника.
3. В чате отправить любую команду, начинающуюся с `/` (например `/id`) — обычные сообщения бот не
   видит по умолчанию (Telegram privacy mode для ботов).
4. Открыть в браузере `https://api.telegram.org/bot<TOKEN>/getUpdates` (подставив токен), в
   JSON-ответе найти `"chat":{"id": -100XXXXXXXXXX}` — это отрицательное число и есть
   `TELEGRAM_CHAT_ID` (у групповых чатов id всегда отрицательный).
5. Добавить `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `SUPABASE_DB_URL` (и `GEMINI_API_KEY` для
   GridBot) в **Settings → Secrets and variables → Actions** репозитория.

### Проверка и локальный запуск

Через GitHub: вкладка **Actions** → workflow «telegram-notify» → **Run workflow** → выбрать режим
вручную. «Ничего не шлём» в логе — нормальное поведение при отсутствии подходящей гонки, не
ошибка.

Локально (осторожно — при заполненных `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` в `.env` сообщение
уйдёт в реальный чат лиги):

```bash
cd scripts/telegram
npm install
node notify.js raceweek   # или deadline / results / remind
```

Подробности и разбор кода — [`scripts/telegram/README.md`](scripts/telegram/README.md),
дизайн-документы — [`docs/superpowers/specs/2026-07-19-telegram-notify-design.md`](docs/superpowers/specs/2026-07-19-telegram-notify-design.md)
и [`docs/superpowers/specs/2026-07-20-autoresults-design.md`](docs/superpowers/specs/2026-07-20-autoresults-design.md).

## 🧱 Стек

| Слой | Технологии |
|---|---|
| Фронтенд | React 18 + TypeScript, Vite, React Router, [@dnd-kit](https://dndkit.com) для drag-and-drop, хостинг — GitHub Pages |
| Бэкенд | [Supabase](https://supabase.com) (Postgres + Auth + Edge Functions), вся защита данных — на RLS |
| Данные F1 | [Jolpica API](https://api.jolpi.ca) (Ergast-совместимый), фолбэк — OpenF1 |
| ИИ-игрок | Gemini API (структурированный JSON), собственный сезон + 3 прошлых сезона через Jolpica |
| Автоматика | GitHub Actions (cron): keep-alive, напоминания, автозабор результатов, GridBot |
| Уведомления | Telegram Bot API → общий чат лиги |
| Бэкап | Экспорт снэпшота в Google Sheets по запросу |

## 💻 Локальный запуск

Фронтенд:

```bash
npm install
npm run dev        # http://localhost:5173, нужен .env с ключами Supabase — см. .env.example
npm run build       # прод-сборка (tsc -b && vite build)
```

Серверные скрипты — самостоятельные пакеты в `scripts/*/`, каждый со своим `package.json` и
подключением напрямую к облачному Postgres (см. [«cloud-direct» выше](#-почему-так-устроено-архитектурные-решения)):

```bash
cd scripts/db && npm install && npm test          # миграции + тесты (RLS, формула очков)
cd scripts/import && npm install && npm run all    # импорт пилотов/календаря из Jolpica
cd scripts/ai-player && npm install && npm run predict
cd scripts/telegram && npm install && node notify.js deadline
```
