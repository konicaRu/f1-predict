# scripts/db — применение миграций и тесты Фазы 0 (cloud-direct)

Локальный Docker-стек Supabase на машине разработчика недоступен (сеть/реестр),
поэтому миграции применяются и тестируются **напрямую против облачного проекта**
через session pooler. Это инструменты разработки, не часть приложения.

## Настройка
1. В корне проекта в `.env` должна быть строка `SUPABASE_DB_URL=` —
   **Session pooler** из Dashboard → Connect → Session pooler (IPv4),
   спецсимволы пароля URL-экранировать. `.env` в git не коммитится.
2. `cd scripts/db && npm install` (ставит `pg`).

## Команды
- `npm run rebuild` — дроп объектов Фазы 0 + накат всех `supabase/migrations/*.sql` по порядку.
- `npm run test:score` — 7 примеров формулы очков (plan §3).
- `npm run test:view` — view `scores` на идеальном прогнозе (131/10).
- `npm run test:rls` — 5 RLS-сценариев = критерий готовности Фазы 0 (spec §7).
- `npm test` — все тесты подряд.

## Особенности (почему так)
Сеть до пулера флапает. Поэтому:
- `runner.js` применяет миграции **постейтментно с автокоммитом** и проглатывает
  «already exists» при ретрае (если стейтмент успел выполниться до обрыва).
- Тесты гоняются **одним запросом** (CTE или `DO`-блок), вся вставка в транзакции
  с откатом — облако остаётся чистым.
- `rls.test.js` чистит зависшие `idle in transaction` бэкенды между попытками.

`runner.js sql "<query>"` — разовый запрос для диагностики.
